# iOS 构建模式说明

## 构建模式选择

GitHub Actions 支持两种构建模式：

### 1. Release 模式（默认）
- **用途**: 正式发布版本
- **特点**:
  - 代码经过优化，性能更好
  - 体积更小
  - 不包含调试信息
  - 日志可能被优化掉

### 2. Debug 模式
- **用途**: 测试和调试
- **特点**:
  - 包含完整的调试信息
  - 日志更详细
  - 更容易排查问题
  - 体积较大
  - 性能略差

## 推荐使用场景

### 使用 Debug 模式的情况：
1. **排查连接问题** ✅ **推荐**
   - 需要查看详细的日志
   - 需要调试 VPN 连接问题
   - 需要查看节点选择逻辑

2. **开发测试**
   - 功能测试
   - 问题复现

### 使用 Release 模式的情况：
1. **正式发布**
   - 给用户使用的版本
   - 需要最佳性能

2. **性能测试**
   - 测试实际使用性能

## 如何选择构建模式

在 GitHub Actions 中运行 workflow 时：

1. 点击 "Run workflow"
2. 在 "Build mode" 下拉菜单中选择：
   - `debug` - 用于调试和排查问题
   - `release` - 用于正式发布

## 当前建议

**对于排查连接失败问题，强烈建议使用 `debug` 模式**，因为：
- 日志更详细
- 可以看到所有调试信息
- 更容易定位问题

## 构建命令对比

### Debug 模式
```bash
flutter build ios --debug --no-codesign
```

### Release 模式
```bash
flutter build ios --release --no-codesign
```

## 注意事项

- Debug 模式的 IPA 文件会更大
- Debug 模式可能包含敏感信息，不要分发给最终用户
- Release 模式经过优化，某些调试日志可能不会输出
